## The correction that reshapes the EPIC: perfect separation

**CloudTalk is the sole source of truth for the lead-to-appointment lifecycle, including its own pipeline tags.** We persist NO campaign status locally. On a `meeting_booked` disposition CT hands the lead off to the normal app flow (meeting creation → existing derived customer pipeline).

This was the voip-in-house **2026-05-30 "total separation"** decision, confirmed by the user 2026-06-04. The voip-campaigns plan + a `voipCampaignStatus` enum/mapper had been built on a **local-status-mirror** model that contradicts it. That model is now removed.

## What voip-campaigns persists — the COMPLETE list

1. `voip_campaigns` — CT campaign-id ↔ `source_slug` bridge (+ membership tag, cadence). _(schema file exists, not yet barrel-exported)_
2. `voip_contact_attributes` — CT attribute-id ↔ app-key bridge. _(ditto)_
3. **`voip_campaign_contacts`** — per-customer participation record (renamed from `voip_contact_sync` 2026-06-04): CT contact id + **`voip_campaign_id`** (the ONE campaign the customer is in — a source owns many) + `enrolled_at`/`unenrolled_at` + **`unenroll_reason`** (`graduated|opted_out|disqualified`) + `dial_attempts` + attribute hash + sync metadata. _(ditto)_ "Enrolled now" = row exists AND `unenrolled_at IS NULL`. Membership tag to add/remove is read from the linked `voip_campaigns` row, NOT derived from lead source. **Three exit paths, one idempotent `unenroll(customerId, reason)`** — graduation (meeting), opt-out (STOP), manual disqualification — each reachable from BOTH the UI and a CT webhook disposition (decision #18).
4. `customers` DNC fields (`dncOptedOutAt`, `dncReason`, `dncAddedByUserId`) — shared canonical, already landed.

**That is everything.** `customers` carries NO `voipCampaign*` fields (the dial counter was moved to `voip_campaign_contacts`). No status column, no enrolled/graduated timestamps on customers, no `lead_intake_error`, no `voip_calls`/`voip_messages` shadow rows.
