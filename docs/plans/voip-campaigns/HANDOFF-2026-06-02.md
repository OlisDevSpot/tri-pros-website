### Step 1 — provider restructure (CONFORMANCE FIX)

The new `client-is-the-superset-entry-point` convention requires ONE `<provider>Client` singleton with every action as a method. My earlier `lib/{contacts, campaigns, calls, sms, bulks, attributes}.ts` action files violated this. Restructured to match `twilio/`:

```
providers/cloudtalk/
  client.ts                 ← cloudtalkClient singleton with ~15 methods + verifyWebhookSecret
                              + CloudtalkApiError + CloudtalkResponseValidationError
  types.ts                  ← consumer-shaped domain types (CloudtalkContact, CloudtalkCall,
                              CloudtalkContactSummary) + z.infer<> re-exports of raw wire types
  constants/index.ts        ← HOSTS, RATE_LIMIT thresholds, BULK_MAX cap, cloudtalkTagNames,
                              cloudtalkDispositions, cloudtalkContactAttributeAppKeys, cloudtalkPricing
  schemas/                  ← outbound-API zod (what WE send) — unchanged from prior session
    primitives.ts contact.ts call.ts sms.ts campaign.ts bulk.ts
  webhooks/events.ts        ← inbound-payload zod, 5-event discriminated union
                              (renamed from webhooks/types.ts)
  DOCS.md                   ← NEW — superset-client pattern + invariants + quirks table
  README.md                 ← refreshed — history + tooling reference only
```

**Deleted:** `lib/{attributes,bulks,calls,campaigns,contacts,sms,types}.ts`, `webhooks/verify.ts`, `webhooks/lifecycle-mapper.ts` (moved), `constants/pricing.ts` (merged into `constants/index.ts`).

**Moved:** `webhooks/lifecycle-mapper.ts` → [`src/shared/services/voip/campaigns/lifecycle-mapper.ts`](../../../src/shared/services/voip/campaigns/lifecycle-mapper.ts). Reason: it does app-domain translation (CT disposition → `VoipCampaignStatus`); providers don't carry domain types per `providers-have-no-domain-types-in-signatures`.

## Open decision (gating Section C — needs your call)

**Where does the attempt counter live?** CT does NOT fire an exhausted webhook; we count `call.ended` events ourselves and emit `cadence_exhausted` when count hits `voip_campaigns.attempts_per_contact` (default 10).

Two options:

**(a) Counter column on `customers`** — RECOMMENDED for Phase 1 simplicity:
```ts
voipCampaignAttempts: integer('voip_campaign_attempts').notNull().default(0),
```
- 1 new column on existing table
- Atomic UPDATE in `lifecycle.recordAttempt`
- Resets to 0 on re-enrollment
- No per-attempt audit trail; if we need one later, add the table without breaking this

**(b) New `voip_campaign_attempts` event-log table:**
```sql
CREATE TABLE voip_campaign_attempts (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  customer_id uuid NOT NULL REFERENCES customers(id) ON DELETE CASCADE,
  ct_call_uuid text NOT NULL UNIQUE,    -- idempotency
  is_voicemail boolean NOT NULL,
  ended_at timestamptz NOT NULL,
  created_at timestamptz NOT NULL DEFAULT now()
);
CREATE INDEX voip_campaign_attempts_customer_idx ON voip_campaign_attempts (customer_id);
```
- Per-attempt audit trail (useful for support/debugging "why was this contact exhausted?")
- UNIQUE on `ct_call_uuid` gives idempotency on webhook re-delivery
- Count is `SELECT COUNT(*) WHERE customer_id = ?`
- More schema cost

**User signaled option (a) before the interrupt.** Next session: confirm `(a)`, add the column to `customers.ts`, drop into Section C.

2. **`src/shared/services/voip/campaigns/lifecycle.service.ts`** — the core orchestration
   ```ts
   lifecycleService.applyEngagement(ctx, { ctContactId, callUuid }): DalReturn<void>
     - if status === 'lead' → transition to 'engaged'
     - push removeTags(['Lead']) + addTags(['Engaged']) to CT
     - idempotent: no-op if already engaged or terminal

   lifecycleService.recordAttempt(ctx, { ctContactId, isVoicemail, endedAt }): DalReturn<void>
     - increment voipCampaignAttempts (option a) OR insert voip_campaign_attempts row (option b)
     - load voip_campaigns by source_slug → check attempts_per_contact cap
     - if reached + status not terminal → transition to 'exhausted'
     - push removeTags(['Lead', 'Engaged']) + addTags(['Exhausted']); KEEP Campaign-X

   lifecycleService.applyDisposition(ctx, { ctContactId, disposition, note }): DalReturn<void>
     - load current status; call mapDispositionToStatus(currentStatus, disposition)
     - guard: no-op if mapped status === currentStatus (race-defense vs. Call.Modified re-fires)
     - persist new status
     - push tag swap matching new status (with Campaign-X removal on terminal Booked/DoNotCall)
   ```
   All push-to-CT calls go through `cloudtalkClient.{addTags, removeTags}`. Mutations live in `entities/customers/dal/server/mutations.ts` (per `services-orchestrate-dal-implements` rule — no raw `db.update` in services).
