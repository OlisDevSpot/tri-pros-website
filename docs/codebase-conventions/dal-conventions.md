# Data Access Layer (DAL) Conventions

The DAL is the only layer that imports `db`. Everything that reads or writes Postgres flows through here — tRPC procedures, services, jobs, scripts, RSC. DAL functions are framework-agnostic: they receive a `ScopedContext` and return `DalReturn<T>`, never throwing on domain errors.

DAL pattern adapted from [WebDevSimplified/next-js-data-access-layer](https://github.com/WebDevSimplified/next-js-data-access-layer).

## Rules

### only-dal-imports-db

Only files under `src/shared/dal/` or `src/shared/entities/*/dal/` import the `db` client from `@/shared/db`. tRPC procedures, services, and jobs always go through DAL.

**Why**: makes db access auditable, reusable, and visibility-scope-correct. Skipping DAL leaks the `isOmni`-or-predicate dance back across the codebase.
**Reference impl**: any file in `src/shared/entities/proposals/dal/server/`
**Enforced by**: convention (compliance sweep tracks violations — see compliance todo in this migration)

### dal-returns-dalreturn

Every exported DAL function returns `Promise<DalReturn<T>>`. Construct with `dalSuccess(data)` / `dalError({ type: 'not-found' })`. Never throw on domain errors (not-found, forbidden, etc.). Use `ThrowableDalError` inside a `dalDbOperation` block when a mid-query check needs to short-circuit.

```ts
export async function getById(
  ctx: ScopedContext,
  input: { id: string },
): Promise<DalReturn<Row<typeof proposals> | undefined>> {
  // ...
}
```

**Why**: callers (tRPC, services, jobs, RSC) need to decide error policy. tRPC maps errors to HTTP via `dalToTrpc()`. Services may retry or escalate. Jobs may log and continue. Throwing forces every caller into try/catch.
**Reference impl**: `src/shared/dal/server/types.ts` (definitions); `src/shared/entities/proposals/dal/server/queries.ts` (usage)
**Enforced by**: tsc (return type annotation is required)

### scoped-context-first-arg

Every DAL function takes `ctx: ScopedContext` as its first argument. `ctx.scope` is the visibility predicate (a Drizzle `SQL` fragment or `null`). DAL applies it to the WHERE clause for any query that should be visibility-scoped.

**Why**: visibility scoping is enforced by the DAL itself, not the caller. Caller can't forget.
**Reference impl**: `src/shared/dal/server/lib/create-crud-dal.ts`
**Enforced by**: convention (no machine enforcement yet)

### two-entry-points-into-dal

Two pathways construct `ScopedContext`:

1. **Via tRPC** — middleware (`scopeMiddleware`, `shareableMiddleware`) resolves session, ability, and scope from the HTTP request and forwards `AuthedContext` (a narrowed `ScopedContext`) to the procedure body.

2. **Via server-initiated work** (jobs, webhooks, cron, RSC) — caller imports `SYSTEM_CONTEXT` (full access, scope = null) or builds a scoped context via a future `buildUserContext()` helper.

**Why**: a job or webhook receiving a payload from QStash needs the same DAL — but with different auth semantics. Same function, different ctx.
**Reference impl**: `src/shared/dal/server/types.ts:SYSTEM_CONTEXT`; jobs in `src/shared/services/providers/upstash/jobs/`
**Enforced by**: convention

### explicit-return-types

Every exported DAL function has an explicit return type annotation. No `any`. Named types live above the function if the return type is a non-trivial composite.

```ts
export type ProposalFullView = { proposal: Row<typeof proposals>, customer: Row<typeof customers> }

export async function getFullView(
  ctx: ScopedContext,
  input: { id: string, token?: string },
): Promise<DalReturn<ProposalFullView | undefined>> {
  // ...
}
```

**Why**: explicit return types catch breaking changes at compile time and make IDE hovers useful.
**Reference impl**: `src/shared/entities/proposals/dal/server/queries.ts`
**Enforced by**: tsc (no implicit any)

### crud-vs-business

DAL is split into two kinds of functions:

- **CRUD DAL** — 5 single-row operations (`getById`, `create`, `update`, `delete`, `duplicate`). Always returns `Row<TTable>`. Generated by `createCrudDal(spec)`. List is NOT CRUD.
- **Business DAL** — entity-specific queries with custom return shapes (enriched reads, multi-table joins, aggregates, list queries with filters). Hand-written. Free-form return types.

Both live under `src/shared/entities/<entity>/dal/server/` (or `src/shared/dal/server/<domain>/` for cross-entity DAL).

**Why**: CRUD is mechanical and benefits from a factory. Business queries are inherently entity-specific — forcing them into a generic single-table factory creates worse code.
**Reference impl**: `src/shared/entities/proposals/dal/server/queries.ts` (mix of CRUD via spec + business reads)
**Enforced by**: convention + ADR-0002

### no-barrel-files-in-dal

DAL directories (`dal/server/`, `dal/client/`) do not have `index.ts` barrel files. Imports use the full path: `import { listProposals } from '@/shared/entities/proposals/dal/server/queries'`.

**Why**: barrels create implicit re-exports that obscure where code lives. Full paths are greppable.
**Reference impl**: `src/shared/entities/proposals/dal/server/` (no index.ts)
**Enforced by**: convention

### client-dal-hook-when-reused

Client-side custom hook wrappers only when the query is reused in 2+ places. For one-off queries, use `useTRPC()` + `useQuery(trpc.x.y.queryOptions(...))` directly in the component.

**Why**: a one-call-site hook is just a rename — pure indirection.
**Reference impl**: `src/shared/dal/client/hooks/`
**Enforced by**: convention

## Lifecycle Hooks

The DAL layer is the hook execution engine for all entity lifecycle hooks. Both `before` and `after` hooks execute inside `createCrudDal` functions.

- Hooks can call other DAL functions (via `SYSTEM_CONTEXT` or passed `ctx`)
- Never use naked `db` in hooks — always go through DAL
- `before` hooks return type: `Promise<T> | T` (sync enrichment still works)
- `after` hooks return type: `Promise<void>` (fire-and-forget decided by hook impl)
- All hooks receive `ScopedContext` — handle `ctx.session` being null for jobs/services

## Anti-patterns

- **Throwing in DAL** (`throw new Error('not found')`). Use `dalError({ type: 'not-found' })`.
- **Inline `db.select()` in a service or tRPC procedure.** Move to DAL.
- **Forgetting `ctx.scope` in a query.** Add `.where(ctx.scope ?? sql\`true\`)` or use `createCrudDal` which applies it automatically.
- **Returning `Promise<T>` without `DalReturn`.** Wrap in `dalSuccess`/`dalError`.
- **Adding `index.ts` to a DAL folder.** Import the file directly.
